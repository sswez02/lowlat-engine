# ADR-002: Event data model uses std::variant

## Status

Accepted

## Context

ADR-001 establishes a single virtual Operator interface so that pipelines can be composed at runtime from mixed operator types.

The next design question is what event type should flow through that interface. The engine needs one common process(...) signature so that the main pipeline loop can call every operator uniformly, while still supporting multiple event shapes across different pipelines.

The event representation should preserve type safety where possible and avoid unnecessary heap allocation in the per-event hot path.

## Decision

Use a closed event model based on std::variant.

The engine event type will be represented as:

```cpp
using Event = std::variant<TickEvent, AudioSample, IMUSample>;
```

Operators will process events through the common signature:

```cpp
void process(const Event& event);
```

Concrete operators will inspect the active event alternative using std::visit or targeted access helpers such as std::get_if.

The specific event alternatives may change as the engine becomes more concrete, but the architectural decision is to use a closed std::variant event type rather than templating the operator interface or using open-ended type erasure.

## Alternatives Considered

### Templated Operator<EventT>

A templated operator interface would make each operator specific to one event type, for example Operator<TickEvent> or Operator<AudioSample>.

This gives strong compile-time type safety and allows the compiler to specialize code for each event type. However, it also makes the event type part of the operator type. Operator<TickEvent> and Operator<AudioSample> are different C++ types.

That conflicts with ADR-001, where the pipeline stores mixed operators behind one common Operator interface. A templated Operator<EventT> would push the design back toward compile-time-specific pipelines and make runtime composition harder.

### Type-erased std::any

A type-erased interface such as process(const std::any&) would preserve one common operator signature and allow any event type to be passed through the pipeline.

This was not chosen because it is too loose for the core event model. std::any can hold almost any type, so wrong event/operator combinations are not rejected at compile time. An operator may only fail later when it attempts an std::any_cast to the expected type. That moves mistakes from compile time to runtime.

### Type-erased void\*

A lower-level type-erased interface such as process(const void\*) would provide maximum flexibility and avoid some of the overhead of std::any.

This was not chosen because it removes too much type information from the engine. Operators would need to manually cast the pointer back to the expected event type. An incorrect cast can lead to undefined behavior. That is not a suitable default for the core event interface.

## Consequences

### Positive

- The engine keeps one shared Event type for the virtual Operator interface.
- Pipelines can continue to use std::vector<std::unique_ptr<Operator>> from ADR-001.
- The valid event alternatives are visible in the type system.
- Invalid event types cannot enter the core event model without updating the Event variant.
- std::variant stores its active alternative inline, so the event wrapper does not require heap allocation solely to erase the event type.
- std::visit gives a structured way to dispatch based on the active event alternative.

### Negative

- The event set is closed. Adding a new event type requires modifying the Event alias.
- Changing the event alternatives requires recompiling code that depends on Event.
- The size of Event is at least the size of its largest alternative plus a discriminator/tag.
- Operators that only care about one event type must still handle or ignore other alternatives.
- The cost of std::visit must be measured once realistic event types and operators exist.

## Validation

Variant dispatch is measured by BM_DirectCallVariantPolymorphic in benchmarks/bench_operator.cpp. ADR-001 documents the full operator dispatch benchmark set; this ADR focuses on the additional cost of the std::variant event model.

BM_DirectCallVariantPolymorphic uses a concrete NoOpOperator and varies the active Event alternative each iteration by indexing into a preconstructed array of four events using input & 3. The array contains std::monostate, TickEvent, AudioSample, and IMUSample.

On the local WSL2 Ubuntu setup, the measured median result was:

- `BM_DirectCallVariantPolymorphic_median`: 0.098 ns Time / 0.100 ns CPU

This is effectively identical to the direct-call baseline from ADR-001:

- `BM_DirectCall_median`: 0.097 ns Time / 0.100 ns CPU

The matching result means this benchmark did not show a separate std::visit cost. The visitor body is simple, and the event alternatives are visible to the compiler in a local const array. Because of that, the compiler can inline and simplify the visit path. In this benchmark, std::visit is effectively optimised away.

This is still relevant to the intended engine design. The engine uses a closed Event variant and operators registered during startup/configuration. That gives the compiler good visibility into the event types and visit paths. In this intended hot path, the measured extra cost of the std::variant wrapper is effectively zero.

This does not mean std::visit is always free. If events came from external code, plugins, or another boundary where the compiler could not reason about the active type, std::visit would likely have a real runtime cost. A reasonable warm-cache estimate for that harder case is roughly 1-3 ns per visit.

Even at the upper estimate of 3 ns per visit, a 3-operator chain would spend roughly 9 ns on visit dispatch. That is about 1.8% of the current 500 ns p50 latency budget. In the measured intended case, the added cost is effectively zero.

This supports the ADR-002 decision. The closed-set std::variant model keeps event types visible in the type system, avoids heap allocation, and adds no measurable cost in the current hot-path benchmark. In less compiler-friendly cases, the expected cost is still small relative to the project latency target.
