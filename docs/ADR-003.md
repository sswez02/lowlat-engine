# ADR-003: Pipelines are single-threaded; parallelism is per-pipeline, not per-stage

## Status

Accepted

## Context

This project is a low-latency C++20 event/stream processing engine. The target latency budget is 500 ns p50 for a short operator chain, or roughly 165 ns per operator in a 3-stage pipeline.

The engine currently targets three example workloads:

- Market ticks, where each event may update indicators such as VWAP, EMA, or signal state.
- Audio DSP, where each sample usually performs a small amount of filter or transform work.
- Telemetry/IMU data, where event rates are lower and per-event work is also small.

These workloads are expected to use light per-event operators. The scheduling question is where threads should live: inside each pipeline between stages, or outside pipelines by running separate pipelines independently.

## Decision

Each pipeline runs on a single thread.

Operators inside a pipeline are called inline, in order, through the operator vector. There are no queues, locks, or thread handoffs between stages inside one pipeline.

Parallelism is achieved by running multiple independent pipelines on different threads.

## Alternatives Considered

### Per-stage threading with lock-free SPSC queues

One option was to give each operator stage its own thread and connect stages with lock-free single-producer/single-consumer queues.

This was not chosen because the queue handoff cost is large relative to the operator budget. A warm queue handoff can cost roughly 50-100 ns due to atomic operations and cache-line movement. In a 3-stage pipeline, two handoffs would cost roughly 100-200 ns before any useful operator work runs. That would consume about 20-40% of the 500 ns p50 budget.

This design can help when each stage is heavy and balanced, but the current workloads have light operators. If one stage is slower than the others, the pipeline is still limited by that slowest stage and queues mainly add latency and backpressure complexity.

### Thread pool with work stealing

Another option was to schedule events or operators onto a shared thread pool.

This was rejected for the hot path. Thread-pool scheduling adds queueing, task packaging, cache movement, and less predictable execution. That model is useful for web servers or asynchronous work, but it is not a good fit for a sub-microsecond event pipeline.

### Hybrid inline/queued operators

A hybrid design would run most operators inline, but move specific expensive operators onto their own thread behind a queue.

This was not chosen for the first implementation because it adds significant complexity before there is evidence that any current operator needs it.

The hybrid model remains a future option if a specific operator becomes expensive enough to justify the queue handoff cost.

## Consequences

### Positive

- No intra-pipeline synchronization cost.
- No queue handoff cost between normal operators.
- The 500 ns p50 latency budget remains available for actual operator work.
- Pipeline execution is simple: iterate over operators and call process(...).
- The design is easier to test, and benchmark.
- The model matches the general low-latency pattern of keeping the hot path single-threaded and predictable.

### Negative

- One slow operator blocks the rest of its pipeline.
- A single pipeline cannot use multiple cores internally.
- Throughput for one pipeline is bounded by the slowest operator in that pipeline.
- Parallelism must come from multiple independent pipelines, not from splitting one pipeline across stages.

## Validation

The current latency budget is 500 ns p50 for a short operator chain. For a 3-stage chain, that gives roughly 165 ns per operator.

A per-stage threaded design would add queue handoffs between operators. With a rough handoff cost of 50-100 ns, two handoffs in a 3-stage pipeline would add about 100-200 ns of overhead. That is about 20-40% of the total 500 ns budget before any operator-specific work has run.

The implemented single-threaded pipeline was benchmarked with two 3-operator cases:

- `BM_PipelineThreeOpNoOp`: 3 NoOpOperator stages processing TickEvents.
- `BM_PipelineThreeOpMixed`: NoOpOperator + IncrementOperator + NoOpOperator processing TickEvents.

The measured median results were:

- `BM_PipelineThreeOpNoOp_median`: 3.94 ns
- `BM_PipelineThreeOpMixed_median`: 4.49 ns

The mixed case is close to 3 times the earlier BM_VirtualDispatchPolymorphic result from ADR-001. That is the expected result for a 3-stage heterogeneous operator chain: each stage performs a real operator call through the pipeline.

The homogeneous NoOpOperator case is slightly cheaper, but still includes the cost of the pipeline loop, vector element access, unique_ptr dereference, and operator call. Compared with the earlier single-operator dispatch benchmark, the extra pipeline overhead is roughly 0.5 ns per stage.

The worst measured 3-stage pipeline result is therefore about 4.49 ns. Against a 500 ns p50 budget, that is about 0.9% of the total budget. This leaves roughly 495 ns for actual operator-specific work.

This supports the ADR-003 decision. Single-threaded inline pipeline dispatch is cheap enough that the pipeline scaffolding is not the limiting factor. A per-stage threaded design would spend far more time on queue handoff and synchronization than the current design spends on the whole 3-operator dispatch path.

This decision should be revisited if a specific operator consistently performs work above roughly 1 microsecond per event. At that scale, a 50-100 ns queue handoff may be acceptable relative to the operator cost. Until such an operator exists and is measured, per-stage threading is not justified.
