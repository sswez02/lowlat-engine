# ADR-001: Operator interface uses virtual dispatch

## Status

Accepted

## Context

This project is a low-latency C++20 event/stream processing engine. The engine needs to compose operators into pipelines, and those pipelines should be configurable at runtime, such as during startup or configuration loading.

The main pipeline loop should not need to know every concrete operator type. For example, a pipeline may contain a filter operator, a transform operator, and a sink operator, but the execution code should only need to treat each stage as an Operator.

The design also needs to support adding new operators without rewriting unrelated pipeline execution code.

## Decision

Use a virtual base class Operator.

Concrete operators implement process(const Event&). The Event type is established in ADR-002.

Pipelines store operators as:

std::vector<std::unique_ptr<Operator>>

This gives the engine one common interface for mixed operator types while still allowing each concrete operator to implement its own processing logic.

## Alternatives Considered

### CRTP / static polymorphism

CRTP would avoid virtual dispatch by letting the compiler resolve calls at compile time. Each concrete operator would inherit from a base class templated on the concrete operator type.

This was not chosen because it makes runtime-composable pipelines harder. The base type becomes tied to the concrete derived type, so different operators do not share one simple runtime base interface. That works well when the operator types are known at compile time, but this engine needs to build mixed operator chains at runtime.

### Fully template-based pipeline

Another option was to template the whole pipeline so the exact operator chain is known at compile time. This would give the compiler maximum information and could allow more inlining.

This was not chosen because it makes the pipeline less flexible at runtime. Each operator chain becomes a different C++ type, so changing the pipeline shape is no longer a simple runtime configuration problem. This engine needs runtime composition more than it needs to remove the virtual dispatch cost, which subsequent benchmarking confirms is acceptable.

## Consequences

### Positive

- Pipelines can hold different concrete operator types in one container.
- Pipeline construction can happen at runtime.
- Adding a new operator does not require changing the main pipeline execution loop.
- The core pipeline code depends only on the Operator interface, not on every concrete operator type.

### Negative

- Virtual dispatch adds an indirect call.
- Operators are stored behind pointers or references rather than directly by value.
- The pipeline must manage operator ownership and lifetime clearly.
- Heap allocation must happen during setup/configuration, not in the per-event hot path.
- The dispatch cost is small but not zero, so it should be measured as the engine becomes more realistic.

## Validation

A benchmark compares direct calls, a devirtualized virtual-looking call, and a polymorphic virtual dispatch call.

The original `BM_VirtualDispatch` benchmark used std::make_unique<NoOpOperator>() directly in scope. Because NoOpOperator is final and the compiler can see the concrete dynamic type, this benchmark can be devirtualized into a direct call. It is useful as a best-case optimized path, but it should not be treated as the real vtable dispatch cost.

`BM_VirtualDispatchPolymorphic` alternates between NoOpOperator and IncrementOperator through a std::vector<std::unique_ptr<Operator>>. This prevents the compiler from proving one fixed dynamic type at the call site and gives a more realistic measurement of polymorphic dispatch.

On the local WSL2 Ubuntu setup, the measured median results were:

- `BM_DirectCall_median`: 0.188 ns Time / 0.198 ns CPU
- `BM_VirtualDispatch_median`: 0.375 ns Time / 0.397 ns CPU
- `BM_VirtualDispatchPolymorphic_median`: 6.65 ns Time / 6.97 ns CPU

The polymorphic dispatch result is the relevant validation number for this decision. At 6.65 ns per dispatch, a 3-operator chain would spend roughly 19.95 ns on dispatch, or about 4% of the current 500 ns p50 latency budget, at least for a warm and predictable benchmark setup.

Real pipelines with colder cache state or less predictable operator types may pay more, so dispatch cost should continue to be measured as the engine becomes more realistic.
